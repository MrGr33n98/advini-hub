**Versão:** 1.0 | **Data:** 25/03/2026 | **Status:** Rascunho para Revisão

## 1. RESUMO EXECUTIVO

O **AdvocaciaHub** é um marketplace B2C/B2B especializado em conectar indivíduos e empresas que buscam serviços jurídicos com advogados e escritórios qualificados. Diferentemente de diretórios genéricos, nossa plataforma foca em **confiança verificável** através de:

- Sistema robusto de avaliações com moderação híbrida (IA + humana)

- Verificação rigorosa de credenciais (OAB, especialidades, histórico disciplinar)

- Matching inteligente baseado em necessidade jurídica específica, não apenas especialidade genérica

- Blog de autoridade com conteúdo produzido por advogados verificados

**MVP Core:** Permitir que usuários busquem advogados por especialidade/localização, visualizem perfis verificados, leiam/escrevam avaliações moderadas e entrem em contato via formulário seguro. Receitas iniciais virão de planos de assinatura para advogados (destaque, leads qualificados) e oportunidades de conteúdo patrocinado no blog.

---

## 2. ANÁLISE DE MERCADO & OPORTUNIDADE

### 2.1 Problema a ser resolvido

- **Falta de confiança:** 68% dos consumidores relatam dificuldade em avaliar a qualidade de serviços jurídicos antes da contratação (Fonte: IBGE 2025, Pesquisa de Serviços Profissionais).

- **Informação assimétrica:** Dados sobre especialidade real, taxa de sucesso em casos similares e experiência prática são difíceis de obter.

- **Processo inefficiente:** Busca atualmente requer múltiplas consultas em Google, redes sociais e indicações pessoais.

### 2.2 Oportunidade

- Mercado jurídico brasileiro: ~1,2 milhão de advogados (OAB 2026), crescimento de 3,2% ao ano.

- 41% dos advogados relatam dificuldade em captar novos clientes de forma consistente (Pesquisa FGV 2025).

- Tendência crescente de consumo de serviços profissionais via plataformas (ex: 99Freitas, GetNinjas) cria expectativa por similaridade no segmento jurídico.

### 2.3 Proposta de Valor

| Para Usuários (Clientes) | Para Advogados/Escritórios |

|--------------------------|----------------------------|

| Busca filtrada por especialidade granular (ex: "Direito Trabalhista - Hora Extra") | Leads qualificados com intenção confirmada |

| Avaliações verificáveis com contexto do caso (sem identidade do cliente) | Perfil profissional destacado com métricas de engajamento |

| Agendamento preliminar de consulta (fase 2) | Sistema de agendamento integrado (fase 2) |

| Conteúdo educacional relevante no blog | Autoridade via publicação de artigos no blog da plataforma |

| Transparência total sobre honorários médios por tipo de serviço | Dashboard de performance com benchmarking regional |

| Área | Funcionalidade | Descrição | Critério de Aceitação |

|------|----------------|-----------|------------------------|

| **Sistema de Confiança** | Avaliações com moderação | Escala 1-5 estrelas + comentário modificado (remove dados pessoais) + campo "Tipo de caso" (seleção) | 100% de avaliações revisadas por moderação híbrida (IA + humana) antes de publicar |

| | Verificação de Credenciais | Checagem automática OAB + alerta para pendências | Bloqueio de perfil se OAB inválida/suspensa (integração com API pública OAB quando disponível) |

| **Blog de Autoridade** | Catálogo de artigos | Busca por tag, autor (advogado verificado), data | Artigos só podem ser publicados por usuários com perfil de advogado verificado |

| | Leitura responsiva | Experiência otimizada para mobile-first | LCP < 2.5s em teste Lighthouse |

| **Banner & Monetização** | Sistema de banners gerenciável | Posições: header (topo), sidebar (lateral), entre posts do blog | Controle via Active Admin: upload, link, datas de início/fim, targeting por página |

| **Controle de Acesso** | Papéis claros | Visitante (busca limitada), Usuário Logado (salvar favoritos, avaliar), Advogado (gerenciar perfil/posts), Admin (tudo) | Acesso negado redireciona para login com mensagem contextual |

| **User** | `id`, `email`, `password_hash`, `role` (visitor, client, lawyer, admin), `created_at`, `last_login_at`, `is_active` | 1:0-1 com Advogado (se role=lawyer), 1:0-1 com ClienteProfile (futuro) | Role controla acesso; autenticação via Devise/JWT |

## 6. ARQUITETURA TÉCNICA & REUTILIZAÇÃO DO AS-EDS

### 6.1 Stack Planejada (Espelhando Projeto Atual)

| Camada | Tecnologia | Justificativa |

|--------|------------|---------------|

| **Frontend** | Next.js 14 (App Router), React 18, TypeScript | SSR/SSG otimizado para SEO; compatível com Lovable |

| **Estilização** | Tailwind CSS + AS-EDS (Claymorphism, tokens, componentes shadcn/ui adaptados) | Consistência visual com sistema existente; claymismo transmite modernidade e confiança |

| **Estado** | React Context API + Zustand (para estado global complexo) | Evita prop-drilling; manageable para MVP |

| **Backend** | Ruby on Rails 7.1 (API-only), PostgreSQL, Redis | Maturidade, ecossistema rico (Devise, Pundit, ActiveAdmin), performance comprovada |

| **Autenticação** | Devise + JWT (para API) + sessões web | Padrão Rails sólido; permite autenticação híbrida (web + API) |

| **Admin** | Active Admin (com customizações) | Interface administrativa pronta para gestão de conteúdo, usuários, banners |

| **Deploy** | Docker + AWS (ECS/Fargate ou equivalent) ou Heroku (para MVP inicial) | Escalabilidade; familiaridade da equipe |

### 6.2 Adaptação do AS-EDS para o Setor Jurídico

O AS-EDS será mantido em sua essência (claymorphism, padrão de componentes, tokens base), com adaptações estratégicas para transmitir **confiança, seriedade e acessibilidade** no contexto jurídico:

#### Tokens de Cor (Ajustes Sugeridos)

| Token AS-EDS Original | Valor Original | Valor Proposto (AdvocaciaHub) | Justificativa |

|-----------------------|----------------|-------------------------------|---------------|

| `--primary` (Blue 600) | `221 83% 53%` | `221 70% 45%` | Azul mais sóbrio (menos "tech", mais "institucional") |

| `--secondary` (Amber 500) | `38 92% 50%` | `30 80% 45%` | Âmbar mais suave (menos agressivo, mais acolhedor) |

| `--accent` (Emerald 500) | `142 71% 45%` | `142 60% 40%` | Verde profissional (associado a crescimento justo) |

| `--background` | `222 47% 95%` | `222 40% 98%` | Fundo ligeiramente mais escuro para melhor legibilidade em textos longos (blog) |

| `--foreground` | `222 47% 11%` | `222 50% 10%` | Texto levemente mais suave para reduzir fadiga visual |

#### Componentes Específicos a Criar (Estendendo AS-EDS)

- `LawyerCard`: Variação de `ClayCard` com badge de verificação OAB, faixa de honorários, ícone de especialidade

- `ReviewFilter`: Componento especializado para filtros de avaliação (por tipo de caso, resultado, data)

- `BlogPostCard`: Variação com leitura de tempo estimado, tag de categoria visual

- `SpecialtyBadge`: Badge pequeno para exibir especialidades em listas (com hover para ver descrição completa)

- `VerificationBanner`: Banner discreto no perfil indicando status de verificação (pendente, aprovado, rejeitado)

#### Padrões de Componente a Preservar

- Todos os componentes `clay-*` (botões, inputs, cards, chips, painéis)

- Sistema de variantes (ex: `.clay-btn-primary`, `.clay-btn-outline`)

- Efeitos de hover e transição suaves (`smooth-transition`)

- Uso de `.glass` para modais e dropdowns (mantendo hierarquia visual)

- Classes de utilitário safe-area para mobile

### 6.3 Integração Frontend-Lovable → Backend Rails

O Lovable gerará o código frontend React/Next.js que consumirá uma API RESTful padronizada. Endpoints esperados:

| Recurso | GET (Lista) | GET (Item) | POST | PUT/PATCH | DELETE |

|---------|-------------|------------|------|-----------|--------|

| `/advogados` | Listar com filtros | Obter detalhes | - | - | - |

| `/avaliacoes` | Listar (por advogado) | Obter detalhes | Criar (cliente logado) | - | - |

| `/posts` | Listar (blog) | Obter detalhe | Criar (advogado verificado) | Atualizar (próprio post) | Excluir (próprio post) |

| `/banners/ativos` | Listar por posição | - | - | - | - |

| `/especialidades` | Listar árvore hierárquica | Obter detalhes | - | - | - |

**Autenticação:** Token JWT no header `Authorization: Bearer <token>` para endpoints protegidos. Rotas de auth: `/api/auth/sign_in`, `/api/auth/sign_out`.

**Tratamento de Erros:** Padronizar respostas de erro com:

```json

{

  "error": "validation_failed",

  "message": "Dados inválidos fornecidos",

  "details": { "field": "oab_number", "issue": "formato inválido" }

}

```

### 7.1 Estratégia de Testes por Camada

| Camada | Tipo de Teste | Ferramentas | Meta de Cobertura | Observações |

|--------|---------------|-------------|-------------------|-------------|

| **Frontend** | Unitário de componentes | Vitest + React Testing Library | 85%+ | Foco em componentes complexos (LawyerCard, ReviewFilter, BlogPostCard) |

| | Integração de páginas | Playwright | 70%+ | Fluxos críticos: busca → perfil → avaliação |

| | E2E de jornada | Playwright (com cenários de usuário) | 60%+ | Jornada completa do usuário final (busca até deixar avaliação) |

| **Backend** | Unitário de models | RSpec | 90%+ | Validações, escopos, métodos de negócio |

| | Unitário de services | RSpec | 85%+ | Lógica de matching, cálculo de reputação, moderação de IA |

| | Integração de API | RSpec + Rails request specs | 80%+ | Endpoints com autenticação, validações, respostas |

| | Testes de contrato | Pact (opcional Fase 2) | - | Garantir compatibilidade frontend-backend |

### 7.2 Fluxo de Trabalho TDD

1. **Escreva o teste falhando** (ex: "Usuário deve poder filtrar advogados por especialidade tributária")

2. **Implemente o mínimo para passar** (ex: endpoint de filtro + filtro no frontend)

3. **Refatore com segurança** (ex: extrair lógica de filtro para service reutilizável)

4. **Repita** para próxima requisito

### 7.3 Exemplos de Testes Críticos

#### Frontend (Vitest + RTL)

```ts

test('LawyerCard exibe badge de verificação quando OAB válida', () => {

  const { getByText } = render(<LawyerCard lawyer={mockVerifiedLawyer} />);

  expect(getByText('OAB Verificada')).toHaveClass('clay-badge-success');

});

test('ReviewFilter atualiza URL ao mudar filtro de tipo de caso', () => {

  render(<ReviewFilter />);

  fireEvent.change(getByLabelText('Tipo de caso'), { target: { value: 'trabalhista' } });

  expect(window.location.search).toContain('case_type=trabalhista');

});

```

#### Backend (RSpec)

```ruby

describe Advogado, type: :model do

  it 'é inválido sem número da OAB' do

    advogado = Advogado.new(oab_number: nil)

    expect(advogado).not_to be_valid

    expect(advogado.errors[:oab_number]).to include("não pode ficar em branco")

  end

  it 'calcula média de avaliações corretamente' do

    advogado = create(:advogado)

    create(:avaliacao, advogado: advogado, rating: 4)

    create(:avaliacao, advogado: advogado, rating: 5)

    expect(advogado.avg_rating).to eq(4.5)

  end

end

```

## 8. REQUISITOS NÃO-FUNCIONAIS

### 8.1 Desempenho

- **LCP (Largest Contentful Paint):** < 2.5s em mobile 3G (simulado)

- **TTI (Time to Interactive):** < 3.5s em condições semelhantes

- **Taxa de erros 5xx:** < 0,1% em produção

- **Tempo de resposta API p95:** < 300ms para endpoints críticos (busca, perfil)

### 8.2 Acessibilidade (WCAG 2.1 AA)

- Navegação totalmente teclável

- Contraste mínimo de 4.5:1 para texto normal, 3:1 para texto grande

- Todos os componentes interativos têm rótulos acessíveis (aria-label, visívelmente ocultos se necessário)

- Suporte a leitores de tela (testado com NVDA/VoiceOver)

- Tamanho de fonte mínimo ajustável (sem quebra de layout até 200% zoom)

### 8.3 Segurança

- Proteção contra OWASP Top 10 (injeção, XSS, CSRF, etc.)

- Sanitização rigorosa de todo conteúdo gerado pelo usuário (avaliações, comentários de blog)

- Rate limiting em endpoints sensíveis (login, envio de avaliação)

- Criptografia de dados sensíveis em repouso (PII, se coletados futuramente)

- Headers de segurança: CSP, HSTS, X-Frame-Options, etc.

### 8.4 Escalabilidade

- Arquitetura pronta para horizontal scaling (stateless frontend/backend)

- Uso inteligente de caching (Redis para: sessões, resultados de busca frequentes, contagem de avaliações)

- Banco de dados otimizado com índices em campos de busca comuns (especialidade, cidade, oab_number)

- Preparação para sharding/particionamento caso atinja escala nacional

## 10. RISCOS & MITIGAÇÕES

| Risco | Probabilidade | Impacto | Estratégia de Mitigação |

|-------|---------------|---------|--------------------------|

| **Baixa confiança inicial em avaliações** | Alta | Alto | Implementar moderação híbrida transparente (exibir "Avaliado por equipe de moderação"); mostrar estatísticas de moderação no rodapé |

| **Baixa produção de conteúdo no blog** | Média | Médio | Incentivar via: créditos de destaque no perfil, acesso antecipado a funcionalidades, programa de "Advogado Autoridade" com benefícios |

| **Resistência de advogados à mudança** | Média | Médio | Onboarding personalizado; demonstração clara de ROI (leads qualificados vs. anúncios genéricos); programa piloto com escritórios respeitados |

## 12. APÊNDICES

### 12.1 Glossário de Termos Jurídicos Relevantes

- **OAB:** Ordem dos Advogados do Brasil (registro profissional obrigatório)

- **Especialidade Área de Atuação:** Classificação CNJ (Conselho Nacional de Justiça) para áreas do direito
